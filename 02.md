### 1. 稀疏矩阵乘法方向错误（严重 Bug）

```c++ name=src/sparse_matrix_gr.cpp url=https://github.com/Entropy-pp/brakedown_gr/blob/b3f0fc631c7dea6c921ef6c36424e55f8b0b18da/src/sparse_matrix_gr.cpp#L38-L45
void sparseMatVecMulAdd(const SparseMatrixGR& M, const ZZ_pE* input, ZZ_pE* target) {
    for (long i = 0; i < M.n_rows; i++) {
        for (const auto& entry : M.rows[i].entries) {
            // target[col] += input[i] * coeff
            target[entry.col] += input[i] * entry.val;
        }
    }
}
```

**问题**：这个函数计算的是 `target = M^T * input`（转置乘法），而不是标准的 `target = M * input`。

- `M` 有 `n_rows` 行 `n_cols` 列，`input` 长度应该是 `n_cols`，`target` 长度应该是 `n_rows`。
- 但你的代码是：`input[i]`（`i` 迭代行）乘以 `M[i][col]`，结果累加到 `target[col]`。
- 这实际上是在计算：`target[col] = Σ_i M[i][col] * input[i]`，即 `M^T * input`。

在 Brakedown 的 forward pass 中，A 矩阵应该是 **压缩** 的：从 `n` 维压缩到 `m` 维（`m < n`），即 `A: n_rows × n_cols`，输入长度 `n_rows = n`，输出长度 `n_cols = m`。但你的 `sparseMatVecMul` 中先 `clear(target[j])` 的范围是 `j < M.n_cols`，然后用转置乘法写入 `target[col]`。

**巧合之处**：由于 `LevelDim` 中 `n = input_dim`、`m = output_dim`，而你的 `SparseMatrixGR` 设 `n_rows = dim.n`、`n_cols = dim.m`，加上你在 `sparseMatVecMul` 中清零的是 `target[j < n_cols]`——这恰好让转置乘法的输出维度对上了 `m`。**所以在当前的代码中，这个"转置"实际上碰巧产生了正确维度的输出。**

**然而**，语义上不正确：注释说 `target[col] += sum_i input[i] * M[i][j]`，这是 `M^T * input`。头文件注释也说 `input has length M.n_rows, target has length M.n_cols`。如果将来 `LevelDim` 的语义发生变化，或者有人按照标准稀疏矩阵乘法的理解来使用这个函数，就会出错。

**建议**：明确注释说明你在计算 `M^T * input`，或者改为标准 `M * input`（输入长 `n_cols`，输出长 `n_rows`），并相应调整调用方。

### 2. Reed-Solomon 求值点选取问题（潜在 Bug）

```c++ name=src/brakedown_code_gr.cpp url=https://github.com/Entropy-pp/brakedown_gr/blob/b3f0fc631c7dea6c921ef6c36424e55f8b0b18da/src/brakedown_code_gr.cpp#L13-L24
static void reed_solomon_gr(const ZZ_pE* input, long input_len,
                            ZZ_pE* output, long output_len)
{
    ZZ_pEX poly;
    for (long i = 0; i < input_len; i++) {
        SetCoeff(poly, i, input[i]);
    }
    for (long i = 0; i < output_len; i++) {
        ZZ_pE pt = indexedElementInExceptionalSet(i + 1);
        output[i] = eval(poly, pt);
    }
}
```

**问题 1**：RS 编码的求值点使用 `indexedElementInExceptionalSet(i + 1)`，即 `{1, 2, 3, ...}`。但 Brakedown 论文中的 RS 编码应使用**足够多的不同求值点**，且这些点需要保证在 Galois Ring 中"足够好"（exceptional set 中的元素差值需要可逆）。

**问题 2**：`output_len` 可能大于 exceptional set 的大小 `2^d`（其中 `d = ZZ_pE::degree()`）。当 `i + 1 >= 2^d` 时，`indexedElementInExceptionalSet` 的实现会截断高位比特（因为循环只跑到 `ZZ_pE::degree()`），导致**求值点重复**，RS 编码退化。

```c++ name=src/gr.cpp url=https://github.com/Entropy-pp/brakedown_gr/blob/b3f0fc631c7dea6c921ef6c36424e55f8b0b18da/src/gr.cpp#L21-L31
ZZ_pE indexedElementInExceptionalSet(long index) {
    ZZ_pX res = ZZ_pX();
    for (int i = 0; i < ZZ_pE::degree(); i++) {
        long mask = 1 << i;
        if ((mask & index) != 0) {
            SetCoeff(res, i);
        }
    }
    return to_ZZ_pE(res);
}
```

当 `index >= 2^degree` 时，高位比特被忽略，例如 `indexedElementInExceptionalSet(1)` 和 `indexedElementInExceptionalSet(1 + 2^degree)` 会返回相同的值。

**建议**：添加断言确保 `output_len < 2^degree`，或者选择更安全的求值策略。

### 3. `brakedown_codeword_len` 计算中丢失最后一个 A 中间结果的长度

```c++ name=include/brakedown_params.h url=https://github.com/Entropy-pp/brakedown_gr/blob/b3f0fc631c7dea6c921ef6c36424e55f8b0b18da/include/brakedown_params.h#L112-L124
inline long brakedown_codeword_len(
    const BrakedownSpec& spec, long log2_q, long n, long n_0)
{
    std::vector<LevelDim> a, b;
    brakedown_dimensions(spec, log2_q, n, n_0, a, b);
    if (a.empty()) return iceil(n * spec.R);
    long total = a[0].n; // first block: original message
    for (size_t i = 0; i + 1 < a.size(); i++) total += a[i].m;
    total += b.back().n; // RS part
    for (size_t i = 0; i < b.size(); i++) total += b[i].m;
    return total;
}
```

**问题**：在 codeword 布局 `[message | A0_out | A1_out | ... | AL-1_out | RS_out | BL-1_out | ... | B0_out]` 中，`brakedown_codeword_len` 通过 `for (size_t i = 0; i + 1 < a.size(); i++) total += a[i].m` 只加了 `A0.m, A1.m, ..., A_{L-2}.m`，**没有加 `A_{L-1}.m`**（最后一个 A 矩阵的输出维度）。

然而在 `brakedown_encode` 中（第 176-184 行），最后一个 intermediate（即 `A_{L-1}` 的输出，长度 `a[L-1].m`）被写入了 codeword。而 `b.back().n = iceil(a[L-1].m * R)` ≠ `a[L-1].m`。

这意味着 **codeword_len 可能不足以容纳所有块**，导致 `brakedown_encode` 中 `offset > cw_len`，写入被截断（因为有 `std::min(sz, cw_len - offset)` 保护），最后的 B outputs 可能部分丢失。

**建议**：修改为：
```c++
for (size_t i = 0; i < a.size(); i++) total += a[i].m;
```
即包含所有 A 输出的维度，同时需重新审视 RS 部分的长度计算。

### 4. Backward pass B 矩阵的输入组合逻辑（设计风险）

```c++ name=src/brakedown_code_gr.cpp url=https://github.com/Entropy-pp/brakedown_gr/blob/b3f0fc631c7dea6c921ef6c36424e55f8b0b18da/src/brakedown_code_gr.cpp#L124-L160
    // ---- Backward pass: B_i expands ----
    std::vector<std::vector<ZZ_pE>> b_outputs(L);
    {
        const auto& B = code.b_mats[L - 1];
        b_outputs[L - 1].resize(B.n_cols);
        for (long j = 0; j < B.n_cols; j++) clear(b_outputs[L - 1][j]);

        long b_input_len = B.n_rows;
        std::vector<ZZ_pE> b_input(b_input_len);
        long copy1 = std::min(rs_input_len, b_input_len);
        for (long j = 0; j < copy1; j++) b_input[j] = intermediates[L - 1][j];
        long copy2 = std::min(rs_output_len, b_input_len - copy1);
        for (long j = 0; j < copy2; j++) b_input[copy1 + j] = rs_output[j];

        sparseMatVecMul(B, b_input.data(), b_outputs[L - 1].data());
        // ...
    }
```

**问题**：根据 Brakedown 论文（参考 [GLSTW21]），backward pass 中 B_L 的输入应该是 RS 编码的**完整输出**（即 `[rs_input || rs_output]` 拼接起来），而不仅仅是截断到 `B.n_rows`。

`B.n_rows = iceil(a[L-1].m * R)`，应该等于 `rs_input_len + rs_output_len`（即 RS 编码后的总长度）。但由于 `rs_output_len = iceil(rs_input_len * R) - rs_input_len`，所以 `rs_input_len + rs_output_len = iceil(rs_input_len * R)`。

如果 `B.n_rows != rs_input_len + rs_output_len`，则输入被截断或有零填充。**建议添加断言验证**：
```c++
assert(B.n_rows == iceil(rs_input_len * code.spec.R));
```

类似地，中间层 B_i 的输入是 `[intermediates[i] || b_outputs[i+1]]`，但 `Bi.n_rows = iceil(a[i].m * R)`。需要验证 `intermediates[i].size() + b_outputs[i+1].size() == Bi.n_rows`。

### 5. 哈希函数安全性（重要警告）


**问题**：你用的是自定义的 FNV-1a 扩展哈希，**完全不具备密码学安全性**。Brakedown PCS 的安全性依赖于 Merkle 树的 collision-resistance，而 FNV-1a 不是抗碰撞的。

对于毕设演示来说这可能可以接受，但必须在论文/报告中声明这一点。代码注释中也已提到了替换方案。

**建议**：使用 OpenSSL 的 SHA-256：
```c++
#include <openssl/sha.h>
```

### 6. Proximity Testing 中随机系数使用 `randomInvertible()` 的合理性

```c++ name=src/brakedown_pcs_gr.cpp url=https://github.com/Entropy-pp/brakedown_gr/blob/b3f0fc631c7dea6c921ef6c36424e55f8b0b18da/src/brakedown_pcs_gr.cpp#L105-L112
    proof.prox_coeffs.resize(num_prox);
    for (long t = 0; t < num_prox; t++) {
        proof.prox_coeffs[t].resize(num_rows);
        for (long i = 0; i < num_rows; i++) {
            proof.prox_coeffs[t][i] = randomInvertible();
        }
        proof.combined_rows.push_back(combine_rows(proof.prox_coeffs[t]));
    }
```

**问题**：Proximity testing 的系数应由 **verifier** 通过 Fiat-Shamir 变换（哈希 transcript）生成，而非 prover 自己随机生成后传给 verifier。

在你的实现中：
1. Prover 自己选择 `prox_coeffs` 并发送给 verifier
2. Verifier 在 `brakedown_verify` 中直接使用 prover 提供的 `prox_coeffs`

**这是不安全的**：恶意 prover 可以选择特殊的系数使检查通过。在交互式协议中，这些系数应由 verifier 发送；在非交互式协议（Fiat-Shamir）中，它们应从 transcript 的哈希推导出来。

**同样，column opening 的位置也应由 Fiat-Shamir 推导**，而非 prover 选择：

```c++ name=src/brakedown_pcs_gr.cpp url=https://github.com/Entropy-pp/brakedown_gr/blob/b3f0fc631c7dea6c921ef6c36424e55f8b0b18da/src/brakedown_pcs_gr.cpp#L125-L127
    for (long t = 0; t < num_open; t++) {
        long col_idx = RandomBnd(code.codeword_len);
        proof.column_indices[t] = col_idx;
```

**建议**：实现一个简单的 Fiat-Shamir transcript：
- Commit 后，hash(root) → 得到 proximity 系数
- Proximity rows 确定后，hash(transcript) → 得到 column indices

### 7. `num_rows == 1` 时的特殊处理


### 8. `brakedown_params.h` 中 `log2_q` 的计算

```c++ name=src/brakedown_code_gr.cpp url=https://github.com/Entropy-pp/brakedown_gr/blob/b3f0fc631c7dea6c921ef6c36424e55f8b0b18da/src/brakedown_code_gr.cpp#L31
    long log2_q = k * degree;
```

**问题**：Galois Ring `GR(2^k, d)` 的大小是 `(2^k)^d = 2^(k*d)`，所以 `log2_q = k * degree` 是正确的。但 Brakedown 的原始论文中，参数 `q` 通常指的是**基域**的大小而非整个环的大小。在 [GLSTW21] 中，proximity testing 的参数计算用的是：

```c++
inline long brakedown_num_proximity_testing(
    const BrakedownSpec& spec, long log2_q, long n, long n_0)
{
    long cw_len = brakedown_codeword_len(spec, log2_q, n, n_0);
    return iceil(spec.LAMBDA / ((double)log2_q - std::log2((double)cw_len)));
}
```

当 `log2_q < log2(cw_len)` 时，分母为负，结果变成负数的 `iceil`。**建议添加检查**：
```c++
assert((double)log2_q > std::log2((double)cw_len));
```

### 10. Codeword 布局中间 intermediates 的排列顺序

在 `brakedown_encode`（第 163-203 行）中，codeword 布局为：

```
[message] [intermediates[0]] [intermediates[1]] ... [intermediates[L-1]] [rs_output] [b_outputs[L-1]] ... [b_outputs[0]]
```

但在 `brakedown_codeword_len` 中：
```
total = a[0].n + sum(a[i].m for i=0..L-2) + b.back().n + sum(b[i].m for i=0..L-1)
```

这里 `a[0].n` = `row_len`（message），`a[i].m`（`i=0..L-2`）= intermediates 0 到 L-2。**缺少 `intermediates[L-1]`**（即 `a[L-1].m`），而 `b.back().n` 是 RS 编码的输入+输出长度。

所以编码和长度计算之间存在**不一致**，与问题 3 是同一个根因。

## 必须修复的问题（影响性能数据的正确性/公平性）

### 🔴 问题 1：`brakedown_codeword_len` 少算了最后一个 intermediate 的长度

这是唯一会导致**程序行为错误**从而**污染性能数据**的 bug。

```c++ name=include/brakedown_params.h url=https://github.com/Entropy-pp/brakedown_gr/blob/b3f0fc631c7dea6c921ef6c36424e55f8b0b18da/include/brakedown_params.h#L112-L124
// 当前代码
long total = a[0].n;
for (size_t i = 0; i + 1 < a.size(); i++) total += a[i].m;  // ← 少了 a[L-1].m
total += b.back().n;
for (size_t i = 0; i < b.size(); i++) total += b[i].m;
```

**为什么影响 benchmark**：`codeword_len` 偏小 → `brakedown_encode` 写入时被 `std::min(sz, cw_len - offset)` 截断 → 部分 B 输出丢失 → encoding 做了无用功（计算了但没写入）→ 同时 verifier 看到的 codeword 是不完整的。

这意味着：
- **Commit 时间偏低**（codeword 短了，Merkle 树叶子少了）
- **编码的线性性实际上可能被破坏**（linearity test 能通过可能是因为截断的部分恰好为 0）
- 对比 Rinocchio 时，Brakedown 的开销被**低估**

**修复**：

```c++ name=include/brakedown_params.h
inline long brakedown_codeword_len(
    const BrakedownSpec& spec, long log2_q, long n, long n_0)
{
    std::vector<LevelDim> a, b;
    brakedown_dimensions(spec, log2_q, n, n_0, a, b);
    if (a.empty()) return iceil(n * spec.R);
    long total = a[0].n;                                    // message
    for (size_t i = 0; i < a.size(); i++) total += a[i].m;  // 所有 A 输出，包括最后一个
    long rs_out = b.back().n - (long)a.back().m;            // RS 额外输出部分
    if (rs_out > 0) total += rs_out;
    for (size_t i = 0; i < b.size(); i++) total += b[i].m;  // 所有 B 输出
    return total;
}
```

或者更简单——直接和 `brakedown_encode` 的布局对齐：

```c++ name=include/brakedown_params.h
inline long brakedown_codeword_len(
    const BrakedownSpec& spec, long log2_q, long n, long n_0)
{
    std::vector<LevelDim> a, b;
    brakedown_dimensions(spec, log2_q, n, n_0, a, b);
    if (a.empty()) return iceil(n * spec.R);

    long total = a[0].n;  // message block
    // intermediates[0] .. intermediates[L-2]
    for (size_t i = 0; i + 1 < a.size(); i++) total += a[i].m;
    // intermediates[L-1] (= RS input)
    total += a.back().m;
    // RS output
    long rs_input_len = a.back().m;
    long rs_output_len = iceil(rs_input_len * spec.R) - rs_input_len;
    if (rs_output_len < 1) rs_output_len = 1;
    total += rs_output_len;
    // B outputs
    for (size_t i = 0; i < b.size(); i++) total += b[i].m;
    return total;
```

### 🔴 问题 2：RS 求值点在 `index >= 2^degree` 时重复

**为什么影响 benchmark**：如果你的 `row_len` 较小（比如 ≤ 20），递归停止后 RS 编码的 `output_len` 很小，不太会越界。但当 `row_len` 较大且递归层数少时，RS 底层的 `input_len` 可能接近 `n_0 ≈ 20`，`output_len` = `iceil(20 * 1.72) - 20 ≈ 15`，总共 35 个点，如果 `degree = 8` 则 `2^8 = 256`，够用。

**但如果 `degree` 较小**（比如 `degree = 4`，`2^4 = 16`），而 RS 需要超过 16 个不同的求值点，就会重复，RS 编码退化。

**简单修复**——加个保护断言：

```c++ name=src/brakedown_code_gr.cpp
static void reed_solomon_gr(const ZZ_pE* input, long input_len,
                            ZZ_pE* output, long output_len)
{
    // 确保求值点不会重复
    assert(output_len < (1L << ZZ_pE::degree()));

    ZZ_pEX poly;
    for (long i = 0; i < input_len; i++) {
        SetCoeff(poly, i, input[i]);
    }
    for (long i = 0; i < output_len; i++) {
        ZZ_pE pt = indexedElementInExceptionalSet(i + 1);
        output[i] = eval(poly, pt);
    }
}
```

这样如果参数不对会直接崩溃而不是默默产生错误数据。你的测试用的 `k=64, degree=8`，`2^8=256` 完全够用，所以这个 assert 不会触发。

```c++ name=include/brakedown_params.h
inline long brakedown_codeword_len(
    const BrakedownSpec& spec, long log2_q, long n, long n_0)
{
    std::vector<LevelDim> a, b;
    brakedown_dimensions(spec, log2_q, n, n_0, a, b);
    if (a.empty()) return iceil(n * spec.R);

    // 与 brakedown_encode 的 assemble 布局完全对齐：
    // [message] [inter[0]] ... [inter[L-2]] [inter[L-1]] [rs_output] [b_out[L-1]] ... [b_out[0]]

    long total = a[0].n;  // message

    // 所有 A 中间结果 (inter[0] .. inter[L-1])
```
